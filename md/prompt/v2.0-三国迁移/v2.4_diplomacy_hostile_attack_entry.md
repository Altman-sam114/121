# v2.4 外交敌对攻击入口兼容层提示词

## 1. 目标

继续推进三国多势力、武将、道路和交战迁移：此前 UI 只读预览和地图/面板显示已按 `DiplomaticRelation.status` 区分敌对，但玩家点击攻击、武将宏观进攻 affordance、底层攻击校验和 AI 单位目标筛选仍有静态 `Faction.isHostile(to:)` 口径。本轮把实际攻击入口和攻击合法性收口到 `DiplomacyState.isHostile(between:and:)`，避免 UI 显示、玩家提交、AI 指令生成和 `RuleEngine` 校验分叉。

## 2. 范围

- `WWIIHexV0/App/AppContainer.swift`
  - 地图点击自动攻击目标派生。
  - 点击非己方军队时的直接攻击 gate。
  - 攻击高亮。
  - 武将“进攻郡县”按钮可用性 / 原因。
  - `selectedAttackTarget` 和从目标反推己方相邻防区。
- `WWIIHexV0/Rules/CommandValidator.swift`
  - `.attack` 目标 faction 合法性。
- `WWIIHexV0/Rules/CommandExecutor.swift`
  - `executeAttack` 增加防绕过的外交敌对 guard。
- `WWIIHexV0/Rules/RegionCommandValidator.swift`
  - 旧 region attack 兼容校验。
- `WWIIHexV0/Commands/WarCommandExecutor.swift`
  - 单位敌军强度、敌军存在和可见攻击目标筛选。
- `WWIIHexV0/Rules/RegionCombatRules.swift`
  - 区域交战压力派生。

## 3. 实现要求

- 使用现有 `DiplomacyState.isHostile(between:and:)`，保留其缺建档 fallback 到 `Faction.isHostile(to:)` 的兼容策略。
- 玩家可点击攻击、高亮和武将宏观进攻按钮必须与 `CommandValidator.validateAttack` 口径一致。
- `WarCommandExecutor` 不应再把外交非敌对单位作为攻击目标或敌军强度来源。
- `CommandExecutor.executeAttack` 即使被绕过 `RuleEngine` 直接调用，也不应结算外交非敌对目标。

## 4. 禁止项

- 不修改 `Faction.isHostile(to:)` 语义。
- 不实现完整借道、同盟通行、共同作战堆叠、补给共享或外交宣战制度。
- 不修改移动、伤害、反击、撤退、占领、道路修缮、粮道补给或部署接触规则。
- 不把 `HexTile.controller` / `RegionNode.controller` 的非己方控制、敌控地格排序或部署压力等同改成外交关系；这些仍表示领土控制或战术控制权。
- 不新增持久字段，不修改 JSON schema，不改 Xcode project。
- 不运行本机 Xcode / XCTest / 模拟器 / Probe / Smoke / Stage Regression / Dynamic Theater Regression / Full / 性能测试。

## 6. 验收标准

- 玩家点击攻击、攻击高亮和武将宏观进攻目标派生与 `CommandValidator.validateAttack` 使用同一外交 hostile 口径。
- AI 单位攻击目标筛选不会选择外交非敌对单位。
- 区域交战压力只统计外交 hostile / atWar 单位。
- 缺外交建档的旧数据仍能 fallback 到 `Faction.isHostile(to:)`。
- 本轮不改变移动、道路、补给、部署、占领或完整外交制度。
