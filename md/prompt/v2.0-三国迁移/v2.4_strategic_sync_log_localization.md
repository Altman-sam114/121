# v2.4 战略同步日志三国化兼容层

## 目标

继续推进三国迁移中“武将、道路、交战”相关的可见审计闭环，把占领、动态方面推进、前线变化和战略状态补建时写入 `GameState.eventLog` 的英文 / raw id 文案改成中文三国语义。

本轮聚焦战略同步日志，不改规则计算。

## 范围

- `StrategicStateSynchronizer` 的 region controller 变化事件。
- `WarCommandExecutor` 的防区指令拒绝、region controller 变化、动态方面推进和前线变化事件。
- `CommandExecutor` 普通命令路径的动态方面推进事件。
- `StrategicStateBootstrapper` 的外交、方面、前线、防区补建事件。
- 同步更新 README、核心流程文档、流程图索引、prompt 索引和 update log。

## 非目标

- 不重命名 `RegionId`、`TheaterId`、`FrontZoneId`、`Division`、`Faction.germany/allies` 或任何 Codable rawValue。
- 不改变 `hexToTheater`、`hexToFrontZone`、`regionToTheater` 的权威边界。
- 不改变移动、攻击、占领、修路、外交、生产、胜负、补给、围城、撤退、伤害或 AI 选兵规则。
- 不改 UI / SpriteKit 展示布局。
- 不改 Xcode project。

## 架构依据

- Hex 是战术权威；占领先落到 `HexTile.controller`。
- Region 只从 hex 聚合 controller。
- `regionToTheater` 是初始/基础战区；运行时推进只能改 `hexToTheater`。
- `hexToFrontZone` 是部署层动态归属权威。
- 玩家、AI、武将面板和后续聊天命令都必须经 `Command` / `ZoneDirective -> WarCommandExecutor -> RuleEngine`。

## 实现要求

1. 日志优先使用 `RegionNode.name`、`TheaterNode.name`、`FrontZone.name`。
2. 名称缺失、空白或等于 raw id 时，使用“未命名郡县 / 动态方面”等中文占位，不把 raw id 当玩家文案 fallback。
3. 防区指令拒绝日志中的命令摘要用“进军命令 / 交战命令 / 修缮道路命令”等粗粒度中文描述，避免 `command.displayName` 暴露单位或郡县 raw id。
4. 战略状态补建日志使用“外交档案 / 方面态势 / 前线 / 防区部署”术语。
5. helper 保持在 Rules / Commands 层本地私有实现，不依赖 SwiftUI、SpriteKit 或 `MapDisplayAdapter`。

## 验收标准

- 占领同步、动态方面推进、前线变化和启动补建事件不再出现上述旧英文固定文案或 raw theater id 拼接。
- 事件日志仍保持原 category 与 relatedRecordId。
- 规则状态变化和 Codable schema 不变。
- 文档与 prompt 索引同步。

## 遗留风险

- `Command.displayName`、`CommandIntentAdapter`、`WarDirective` 解码错误和更多 AI 诊断文本仍有 raw id / 英文语义残留，需后续切片继续处理。
- 本轮不跑本机重测试，运行时显示效果等待云端 CI、Agent C artifact 复判或人工授权专项检查。
