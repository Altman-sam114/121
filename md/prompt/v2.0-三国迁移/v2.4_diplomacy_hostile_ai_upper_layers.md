# v2.4 外交敌对 AI 上游摘要兼容层提示词

## 1. 目标

继续围绕武将、道路和交战迁移，把 `ZoneCommanderAgent`、`MarshalBattlefieldSummarizer`、`RulerStrategicSnapshot` 和 `StrategistBattlefieldSnapshot` 的单位级 hostile 摘要从静态二元 `Faction.isHostile(to:)` 迁移到 `GameState.diplomacyState.isHostile(between:and:)`。

本轮重点是 AI 上游摘要和显示分类：

- `ZoneCommanderAgent` 的可见敌军强度、敌军存在和 fallback 防区判断。
- `MarshalBattlefieldSummarizer` 的敌军强度、敌军存在、敌对 objective 名称分类。
- `RulerStrategicSnapshot` 的相邻防区敌军强度统计。
- `StrategistBattlefieldSnapshot` 的邻近 region 敌对单位存在判断。

## 2. 非目标

- 不修改 `ZoneDirective`、`TheaterDirective`、marshal summary Codable schema 或任何 public command schema。
- 不改变 `Command` / `ZoneDirective -> WarCommandExecutor -> RuleEngine` 执行链。
- 不实现完整借道、同盟通行、共同作战堆叠、共享补给或外交宣战制度。
- 不把 `region.controller != faction`、`tile.controller == faction`、`dynamicRegionTouchesZone`、front/deploy topology 等控制权/占领/拓扑事实改成外交判断。
- 不恢复 organization，不重命名 `Division`、`Faction.germany/allies` 等兼容源码名。

## 3. 架构依据

- `HexTile.controller` 和 `Division.coord` 仍是战术权威。
- Region / Theater / FrontLine / WarDeployment 是从 hex、controller、单位位置和动态归属派生的战略层。
- `DiplomacyState.isHostile(between:and:)` 通过双方 primary country 查询 `DiplomaticRelation.status`；`.hostile` / `.atWar` 为敌对，`.neutral` / `.allied` / `.coBelligerent` 为非敌对，缺建档时 fallback 到 `Faction.isHostile(to:)`。
- AI 上游可以用外交 hostile 过滤单位级敌军强度、敌军存在和 objective hostile 展示分类；但 objective/controller 来源仍来自 tile controller 或 region/controller 快照。

## 7. 验收标准

- 上述 AI 上游单位级 hostile 摘要全部使用 `DiplomacyState`。
- objective 和 region controller 来源仍保持 controller / occupation 事实，只在 hostile 分类时使用外交关系。
- directive schema、marshal summary schema 和命令执行链无变化。
- 文档不再把 `ZoneCommanderAgent`、`StrategistAgent`、`RulerAgent` 的单位级 AI hostile 摘要列为待办。
- 轻量检查通过；push 到 `origin/main` 后等待 GitHub Actions `ci-results` workflow 和 Agent C artifact 复判。

## 8. 风险提示

- 本轮不验证运行时 AI 行为，只做轻量语法和文本检查。
- 如果完整多势力外交制度、借道、同盟通行或共享补给尚未实现，不得在文档中写成已完成。
- 如果后续发现 controller / topology 语义与外交 hostile 语义混用，应退回小切片修复，不能回滚整轮。
