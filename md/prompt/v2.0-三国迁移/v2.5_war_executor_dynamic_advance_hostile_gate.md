# v2.5 宏观动态方面推进外交敌对 gate 提示词

## 目标

继续推进武将、道路、交战兼容层，把 `WarCommandExecutor` 宏观军令移动后的动态方面推进从“目标不是己方”收敛为“目标是外交 hostile / atWar”，避免非敌对第三方、中立或借道状态被宏观执行器误判为可推进动态方面。

## 范围

- 修改 `WWIIHexV0/Commands/WarCommandExecutor.swift`。
- 更新 `md/flow/flow.md`、`md/flow/flowchart.md`、`md/prompt/README.md` 和 `update_log.md`。

## 非目标

- 不改变 `CommandExecutor` 直接命令路径；该路径仍作为后续 parity 风险记录。
- 不改变 `OccupationRules`、移动合法性、占领规则、`TheaterSystem`、`WarDeploymentManager`、JSON schema、UI 布局或测试 fixture。
- 不新增真实借道/同盟通行、共同作战堆叠或共享补给制度。
- 不运行本机 Xcode / XCTest / 模拟器 / Probe / Smoke / Stage Regression / Dynamic Theater Regression / Full / 性能测试 / app 启动 / 全量构建。

## 架构依据

- `WarCommandExecutor` 是 AI 防区指令和玩家武将宏观军令落到具体 `Command` 的执行器。
- `OccupationRules.canOccupy` 已按外交 hostile / atWar gate 收束真实占领。
- 文档当前要求 AI / 玩家宏观军令落点敌控优先级、执行器目标筛选和可占领动态推进都优先使用 `DiplomacyState`，不能只按二元 faction 或 “not self” 判断。

## 实现要求

1. `applyStrategicAdvance` 中用于 `TheaterSystem.expandDynamicTheater` 的 faction 必须从 `advancingZoneId` 对应 `FrontZone.faction` 取得；缺失时不推进，不 fallback 到 `.germany`。
2. `shouldAdvanceDynamicTheater` 检查目标已有 zone faction 时，使用 `state.diplomacyState.isHostile(between:destinationFaction,and:advancingFaction)`。
3. `shouldAdvanceDynamicTheater` 检查目标 hex controller 时，同样使用外交 hostile gate。
4. 保持动态方面只推进单个 breakthrough hex，不改变 regionToTheater 或整区归属。

## 验收标准

- `WarCommandExecutor` 宏观动态方面推进不再 fallback `.germany`。
- 宏观动态方面推进只在目标 zone faction 或 tile controller 与推进方外交 hostile / atWar 时触发。
- `CommandExecutor` 直接命令路径差异被记录为后续风险，不在本轮伪装完成。
