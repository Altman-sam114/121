# v2.4 WarCommandExecutor 目标落点 hostile controller 外交化

## 目标

继续围绕武将、道路和交战迁移，把 `WarCommandExecutor` 在宏观军令目标郡县与接近落点排序中的“敌控格优先”判断，从旧二元 `Faction.isHostile(to:)` 改为 `DiplomacyState.isHostile(between:and:)`。

## 范围

- 修改 `WWIIHexV0/Commands/WarCommandExecutor.swift`：
  - `tacticalDestination(in:for:state:)` 的目标郡县候选 hex 排序。
  - `approachDestination(toward:for:state:)` 的接近候选 hex 排序。
- 同步 `README.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`md/prompt/README.md` 和 `update_log.md`。

## 非目标

- 不改变 `CommandValidator`、`RuleEngine`、`MovementRules`、`OccupationRules` 或 `SupplyRules`。
- 不实现完整借道、共同作战堆叠或共享补给。
- 不改变攻击伤害、撤退、占领、动态战区推进或目标 region 选择。
- 不重命名 `Faction`、`Division`、`Theater`、`FrontZone` 等兼容类型。

## 架构依据

- 玩家、AI 和武将宏观军令仍必须收口到 `WarCommandExecutor -> Command -> RuleEngine`。
- `WarCommandExecutor` 只负责把 `ZoneDirective` 降级成具体命令；真实移动、攻击、占领和动态归属仍由规则系统执行。
- 敌对分类优先使用 `DiplomacyState`，缺外交建档时由其默认 fallback 到 `Faction.isHostile(to:)`。

## 实现步骤

1. 阅读 `AGENTS.md`、`update_log.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`md/test/test.md`、总提示词和相关源码。
2. 在 `WarCommandExecutor` 添加私有 helper：
   `isHostileControlled(_:for:in:)`。
3. 在 `tacticalDestination` 和 `approachDestination` 的排序逻辑中，用该 helper 替换 `tile.controller?.isHostile(to: division.faction)`。
4. 保持 `movementRules.shortestPath`、占位过滤、距离排序和坐标稳定排序不变。
5. 更新阶段文档和核心流程文档，说明这是落点优先级外交化，不是完整通行或占领制度。

## 验收标准

- `WarCommandExecutor.swift` 内不再有 `controller?.isHostile(to: division.faction)`。
- 目标郡县候选和接近候选排序都通过 `DiplomacyState` 判断 hostile controller。
- 宏观军令仍经 `CommandValidator -> RuleEngine` 做真实合法性校验。
- 文档明确本轮只改变落点优先级，不改变通行、占领、伤害或动态战区规则。

## 风险

- 本轮会改变外交非敌对控制格在宏观军令落点排序中的优先级；复杂多势力战局中的体感差异需要云端 CI、后续 Agent C 或人工专项复判。
