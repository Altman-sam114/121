# v2.4 外交敌对部署接触兼容层提示词

## 1. 目标

继续推进三国多势力、武将、道路和交战迁移：攻击入口、道路 ZOC、粮道单位阻断、围城邻接和安全补员邻接已接入 `DiplomacyState` hostile / atWar 口径，但 `WarDeploymentManager` 的单位级敌军存在和相邻敌对防区接触仍容易回退到静态 `Faction.isHostile(to:)`。本轮只迁移部署层“单位/防区敌对关系”判断，让中立、停战、同盟或共同作战防区不再误生成敌对部署段或把单位误归前线。

## 2. 范围

- `WWIIHexV0/Rules/WarDeploymentManager.swift`
  - `hasEnemyPresence` 的单位敌对来源。
  - `enemyZoneIdsTouching`、`regionTouchesEnemyZone`、`hexTouchesEnemyZone` 的相邻防区敌对来源。
  - `deploymentRole` fallback 的相邻敌对防区判断。
- 生产调用路径显式传入 `GameState.diplomacyState`：
  - `DataLoader`
  - `StrategicStateBootstrapper`
  - `StrategicStateSynchronizer`
  - `CommandExecutor`
  - `WarCommandExecutor`
  - 部署图层和军队检查器只读显示。

## 3. 实现要求

- 使用 `DiplomacyState.isHostile(between:and:)`，缺外交状态或缺建档时 fallback 到 `Faction.isHostile(to:)`。
- 公开入口用可选默认参数保持历史测试、Probe 和旧调用兼容。
- 有真实 `GameState` 的运行时路径必须显式传入 `state.diplomacyState`。
- 只改敌对关系来源，不改部署层权威结构。

## 4. 禁止项

- 不修改 `Faction.isHostile(to:)` 语义。
- 不把 `HexTile.controller` / `RegionNode.controller` / 非己方控制 hex 或 region 改成外交关系。
- 不改 `hexToFrontZone`、`regionToFrontZone`、`hexToTheater` 推进权威。
- 不改 `isEncircledEnemyContact` 的出口和接触拓扑语义。
- 不实现完整借道、同盟通行、共同作战堆叠、共享补给或外交宣战制度。
- 不修改 JSON schema，不改 Xcode project。
- 不运行本机 Xcode / XCTest / 模拟器 / Probe / Smoke / Stage Regression / Dynamic Theater Regression / Full / 性能测试。

## 6. 验收标准

- 外交非敌对单位不再构成部署层敌军存在。
- 外交非敌对防区不再生成敌对部署段或把单位误归前线。
- 领土 controller、非己方控制 hex / region、encirclement 拓扑、动态 theater/front zone 权威不变。
- 历史测试和 Probe 调用可继续通过默认参数兼容。
- 本机只跑轻量检查，重验证交给 GitHub Actions。
