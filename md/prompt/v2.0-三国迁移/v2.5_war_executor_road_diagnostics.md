# v2.5 宏观军令道路审计提示词

## 目标

继续推进“武将、道路、交战”体验，把 `WarCommandExecutor` 在宏观防区军令中已经使用的有效机动、官道加成和可达性判断写入 `WarDirectiveRecord.diagnostics`，让 AI 与玩家武将宏观军令在 Agent 面板中能解释“为什么这支军队能行军、是否借了官道、到目标郡县是否可达”。

## 范围

- 修改 `WWIIHexV0/Commands/WarCommandExecutor.swift`。
- 修改 `WWIIHexV0/Turn/TurnManager.swift`。
- 修改 `WWIIHexV0/App/AppContainer.swift`。
- 更新 `md/flow/flow.md`、`md/flow/flowchart.md`、`md/prompt/README.md` 和 `update_log.md`。

## 非目标

- 不改变 `MovementRules.shortestPath`、移动成本、敌控区、道路加成、宏观选兵排序、落点排序、攻击、反击、占领或战略同步规则。
- 不改 `WarDirectiveRecord` Codable schema。
- 不改 `AgentPanelView`、`GeneralCommandPanelView` 或 SwiftUI 布局。
- 不运行本机 Xcode / XCTest / 模拟器 / Probe / Smoke / Stage Regression / Dynamic Theater Regression / Full / 性能测试 / app 启动 / 全量构建。

## 架构依据

- `WarCommandExecutor` 已在 `attackSortKey` 中读取 `MovementRules.effectiveMovementLimit`，在 `tacticalDestination` 中用 `MovementRules.shortestPath` 判断目标郡县候选 hex 是否可达。
- `WarDirectiveRecord.diagnostics` 已由 `AgentPanelView` 展示，是承载纯中文审计文本的低冲突入口。
- 玩家武将宏观军令绕过 `TurnManager`，直接在 `AppContainer.submitPlayerDirective` 创建 `WarDirectiveRecord`，因此 AI 和玩家两条记录路径都要合并执行器诊断。

## 实现要求

1. 给非 Codable 的 `WarCommandExecutionResult` 增加 `diagnostics: [String]`。
2. `WarCommandExecutor` 在生成每条宏观命令后、执行前，基于当前 `GameState` 生成道路审计短句。
3. 道路审计包含军队展示名、基础/有效机动、官道机动加成或无加成原因、目标郡县、移动命令可达成本或后备命令说明。
4. 诊断文案必须是纯中文玩家文案，不包含 raw id、JSON、英文 schema 或下划线机器字段，避免 `AgentPanelView.displaySafeDiagnostics` 兜底过滤。
5. 单条 directive 最多展示前三条道路审计，更多命令用“另 N 条道路审计已记录。”汇总，避免面板过长。
6. `TurnManager.executeDirectiveEnvelope` 将 `execution.diagnostics` 合并进 `perDirectiveDiagnostics`。
7. `AppContainer.submitPlayerDirective` 将 `execution.diagnostics` 合并进玩家 `diagnostics`。

## 验收标准

- AI 防区指令记录和玩家武将宏观军令记录都能携带执行器道路审计。
- 道路审计显示有效机动和可达性，但不改变任何命令生成、校验或执行结果。
- Agent 面板可直接展示诊断，不需要 UI 结构变化。
