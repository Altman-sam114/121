# v2.4 太守连通修路兼容层

本文记录“修路不应只是分散补点，应尽量连成官道路段”的小步实现提示词与交付边界。执行前仍必须阅读 `AGENTS.md`、`update_log.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`md/test/test.md` 和总提示词 `md/prompt/v2.0-三国迁移/codex-v2.0-三国aiagent迁移总提示词.md`。

## 目标

- 保持 `Command.improveRoad(regionId:) -> CommandValidator -> CommandExecutor -> EconomyRules` 的统一规则链路不变。
- 保持太守修路资源成本、郡县基础设施 +1、每次最多补两格战术道路的平衡边界。
- 调整 `EconomyRules` 的修路选点：优先从已有官道、邻郡外部官道入口或郡县核心出发，补出短的连续路段。
- 让太守 `roadRepair` 焦点更接近“三国官道/粮道修缮”语义，而不是在郡县内挑两个彼此无关的高分 hex。

## 非目标

- 不实现完整玩家手动修路 UI、道路规划编辑器、跨郡县主干道自动规划或全图路径建设。
- 不改变 `regionToTheater`、`hexToTheater`、`hexToFrontZone`、占领权、军队坐标或战争指令执行规则。
- 不改变资源成本、生产队列、屯田、治安、贡赋、借道或臣属制度。
- 不跑本机 Xcode / XCTest / 模拟器 / Probe / Smoke / Stage Regression / Dynamic Theater Regression / Full。

## 架构边界

```text
GovernorAgent.plan(focus: roadRepair)
  -> TurnManager.governorCommands
  -> Command.improveRoad(regionId)
  -> CommandValidator.validateRoadImprovement
  -> CommandExecutor.executeImproveRoad
  -> EconomyRules.connectedRoadImprovementPlan
  -> MapState.tiles[].hasRoad / RegionNode.infrastructure / EconomyState ledger
```

- 太守层仍不直接改 `GameState`。
- 连通规划只选择当前势力控制、可通行、属于目标郡县的 hex。
- 现有道路、外部相邻道路、代表 hex、城池/关隘和补给源只影响修路优先级，不成为新的权威状态。
- 若找不到连通方案，仍回退到原有高价值目标排序，避免命令无效果。

## 验收标准

- `Command.improveRoad` 仍能经规则层消耗资源、提升基础设施并最多补两格道路。
- 有已有道路时，优先向高价值目标补连通路段。
- 无本郡道路但邻近外部道路时，优先从外部道路入口向内连缀。
- 无任何道路入口时，优先从代表 hex、城池/关隘或补给源等郡县核心起路。
- 文档说明从“最小补两格道路”更新为“连通优先修路”。

## 风险

- 本轮仍不是完整跨郡县主干路规划；每次修路最多两格，复杂地形需要多轮太守行动逐步形成道路。
- 未做运行时 UI 烟测，日志中道路坐标和 AI 面板展示仍待云端 CI、后续 Agent C artifact 复判或人工授权运行检查。
