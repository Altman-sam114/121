# v2.4 交战因素日志审计兼容层

本文记录“交战要能看清楚为什么伤害变化”的小步实现提示词与交付边界。执行前仍必须阅读 `AGENTS.md`、`update_log.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`md/test/test.md` 和总提示词 `md/prompt/v2.0-三国迁移/codex-v2.0-三国aiagent迁移总提示词.md`。

## 目标

- 保持 `CombatRules` 已有伤害公式、撤退、反击和武将修正规则不变。
- 新增只读 `CombatAuditSummary`，把攻击/防御有效值变化、地形、河流、器械攻城、围城、死守和侧击整理为中文日志摘要。
- `CommandExecutor` 在攻击和反击日志中追加交战审计片段，方便 UI、人工和 Agent C 复判“这次交战为什么有加成或减成”。
- 保持交战仍由 `Command.attack -> CommandValidator -> CommandExecutor -> CombatRules` 执行。

## 非目标

- 不新增完整战报面板、技能树、单挑、士气、疲劳或新伤害模型。
- 不改变 `CombatRules.damage`、`attackDamage`、`counterAttackDamage`、撤退或反击规则。
- 不让 `GeneralAgent`、UI 或日志摘要直接修改 `GameState`。
- 不跑本机 Xcode / XCTest / 模拟器 / Probe / Smoke / Stage Regression / Dynamic Theater Regression / Full。

## 架构边界

```text
CombatRules.attackProfile / defenseProfile
  -> CombatRules.effectiveAttack / effectiveDefense
  -> CombatRules.combatAuditSummary
  -> CommandExecutor.combatLog
  -> GameState.eventLog
```

- 交战审计必须和实际 `effectiveAttack` / `effectiveDefense` 使用同一套 profile，不能另写一套会漂移的说明逻辑。
- 武将姓名和攻防修正仍由 `GeneralInfluence.combatSummary` 输出；交战审计负责补充地形、河流、攻城、围城、死守和侧击等通用因素。
- 仅在存在有效值变化、侧击或具体因素时追加日志片段，避免普通平地正面交战产生噪声。
- 日志是审计输出，不参与规则计算，也不作为后续状态权威。

## 验收标准

- 攻击和反击日志在有交战因素时追加 `交战审计：...` 中文片段。
- `effectiveAttack` / `effectiveDefense` 与审计摘要共用同一套 profile，避免规则与解释分叉。
- 交战审计不改变伤害、撤退、反击、占领、动态方面或防区推进。
- 文档说明从“只记录武将影响”更新为“交战因素和武将影响都可审计”。

## 风险

- 本轮只做事件日志审计，不实现发布级战报面板或 UI 图标化解释。
- 未做运行时 UI 烟测，事件日志换行、宽度和战报可读性仍待云端 CI、后续 Agent C artifact 复判或人工授权运行检查。
