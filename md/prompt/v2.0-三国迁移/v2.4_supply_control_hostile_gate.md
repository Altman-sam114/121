# v2.4 SupplyRules 控制格 hostile gate 外交化

## 目标

继续围绕三国迁移中的道路、粮道和交战边界，把 `SupplyRules` 里 capturable 控制格对粮道通行和撤退安全格的阻断判断，从旧二元 `Faction.isHostile(to:)` 迁移到 `DiplomacyState.isHostile(between:and:)`。

## 范围

- 修改 `WWIIHexV0/Rules/SupplyRules.swift`：
  - `isSafeRetreatTile(_:for:in:)` 的控制格阻断。
  - `canSupplyPass(through:tile:for:in:)` 的控制格阻断。
- 同步 `README.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`md/prompt/README.md` 和 `update_log.md`。

## 非目标

- 不实现完整借道制度。
- 不允许 allied / coBelligerent / neutral 单位堆叠。
- 不共享补给源或补给库存。
- 不修改 `MapState.supplySources(for:)`。
- 不修改 `MovementRules` 的非同 faction 单位阻挡。
- 不修改生产、战斗伤害、撤退损耗、围城数值、动态战区推进或 UI 行为。

## 架构依据

- Hex 仍是战术权威，`HexTile.controller` 仍表达真实控制权。
- 敌对分类优先使用 `DiplomacyState`，缺外交建档时由其内部 fallback 到 `Faction.isHostile(to:)`。
- 粮道和撤退仍由 `SupplyRules -> MovementRules -> RuleEngine` 真实规则链路决定，不绕过 `Command` / `RuleEngine`。
- 本轮只改变控制格是否因敌对关系阻断粮道/撤退；单位阻断、ZOC、补给源归属和占领规则保持原边界。

## 实现步骤

1. 阅读 `AGENTS.md`、`update_log.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`md/test/test.md`、总提示词和相关源码。
2. 在 `SupplyRules.isSafeRetreatTile` 中，把 capturable tile controller 判断改为：
   `state.diplomacyState.isHostile(between: controller, and: faction)`。
3. 在 `SupplyRules.canSupplyPass` 中做同样替换。
4. 确认 `state.diplomacyState.isHostile` 保留默认 fallback，避免旧阿登兼容数据缺外交关系时失效。
5. 更新文档，明确：
   - 非敌对控制格不再仅因旧二元阵营关系阻断粮道或撤退安全格。
   - 本轮不实现借道、堆叠、共享补给源或完整外交通行制度。
6. 做轻量检查后 commit 并 push `origin/main`，等待 `ci-results` 云端结果包。

## 验收标准

- `SupplyRules.swift` 内不再有 `controller?.isHostile(to: faction)`。
- 两处控制格 gate 都使用 `DiplomacyState.isHostile(between: controller, and: faction)`。
- hostile 单位阻断和 ZOC 判定仍保留既有规则。
- 文档不再把 `SupplyRules` capturable tile controller 写成继续保留旧阵营 hostile 口径。
- 轻量检查通过，云端 CI 结果包等待 Agent C 核对。

## 风险

- 本轮会改变外交非敌对控制格上的粮道/撤退行为：只要外交关系不是 hostile / atWar，该控制格不再仅因旧二元阵营关系阻断粮道或撤退安全格。
- 没有本机运行时重测试；真实战局表现等待云端构建和后续 Agent C / 人工专项复判。
