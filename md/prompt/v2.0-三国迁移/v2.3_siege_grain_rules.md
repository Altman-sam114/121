# v2.3 围城与粮草最小规则

完成日期：2026-07-05

## 目标

在不扩大 schema、不改 Command / RuleEngine 入口、不引入本机重测试的前提下，把 v2.3 总提示词中的“围城：城池 region 或 fortress hex 被敌邻接且粮道断绝时，防御和恢复下降”落到现有规则系统。

## 范围

- `SupplyRules` 新增围城判定：军队位于城池/关隘 hex，或位于带 city 的 region；该军队无可用补给线；相邻 hex 有敌对军队。
- `CombatRules.effectiveDefense` 对围城守军施加 0.75 倍有效防御修正，最低仍为 1。
- `SupplyRules.applyResupplyRest` 和 `applyEncirclementAttrition` 在围城场景下输出更明确的粮道/围城日志。
- `EconomyRules` 不重复实现恢复下降；既有自动补员已经要求 `supplied`、非撤退、非敌邻，因此围城/断粮单位自然不会自动恢复。

## 非目标

- 不新增 morale、fatigue、grainCarry 字段。
- 不改 `Division`、`Faction`、`SupplyState` 的 Codable schema。
- 不实现多回合围城进度条、城防耐久、投降谈判或单独粮仓库存。
- 不改多势力 turn order。
- 不跑 Xcode / XCTest / 模拟器 / Probe / Smoke / Full。

## 架构依据

- Hex 仍是战术权威：军队位置来自 `Division.coord`，敌邻接和城池/关隘判断都回到 hex。
- Region 只作战略聚合：city region 只用于识别“城池 region”语义，不替代 hex 攻击、移动、占领。
- 玩家和 AI 的攻击、补给、结束回合仍经 `Command -> RuleEngine -> CommandExecutor -> SupplyRules / CombatRules / EconomyRules`。
- 自动补员仍由经济回合结算处理，避免另建一条恢复管线。

## 验收标准

- `SupplyRules.isBesieged` 可解释围城条件，且使用 `Faction.isHostile(to:)` 判断敌对军队。
- 围城防御修正只影响 `CombatRules.effectiveDefense`，不绕过战斗/命令管线。
- 围城/断粮单位不会通过 resupply/rest 或自动补员恢复。
- 核心流程文档和阶段索引同步更新。

## 风险

- 当前只做静态 parse 和文档检查；实际战斗数值、AI 选择和 UI 日志显示仍等待云端 build 与后续 Agent C / 人工复判。
- city region 判定会把 region 内非城池 hex 但处于同一城池 region 的断粮敌邻守军视为围城，这是 v2.3 的可解释简化，后续如需要可收紧到具体 city / fortress hex。
