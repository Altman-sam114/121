# v2.4 军队可达官道预判兼容层

## 1. 目标

在现有“道路机动”预判基础上，让选中军队能看到本回合是否能接入官道网络，以及可达官道中哪些处于敌控区压迫下。该增量继续回应人工强调的“三国要注重武将，做好道路、交战”：道路不只显示郡县总量，也要帮助玩家判断这一支军队当下能否借官道行军、转进或接近粮道。

## 2. 范围

- 修改 `WWIIHexV0/App/AppContainer.swift` 的 `selectedUnitMobilityPreviewNotes` 相关私有派生逻辑。
- 复用 `MovementRules.movementRange` 计算本回合可达格。
- 复用 `MovementRules.isEnemyZoneOfControl` 判断可达官道是否受敌控区压迫。
- 在 `UnitInspectorView` 既有“道路机动”分组里追加只读文字，不改 View 布局。
- 同步 `README.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`md/prompt/README.md` 和 `update_log.md`。

## 3. 非目标和禁止项

- 不修改 `MovementRules` 的移动成本、路径搜索、道路成本、敌控区停止规则或骑兵地形惩罚。
- 不修改 `Command.improveRoad`、`CommandValidator`、`RuleEngine`、太守修路或资源消耗。
- 不新增道路价值评分、完整粮道路径图、道路修建路线预览、AI 自动行军建议或跨郡县交通网络分析。
- 不让 UI 预判写入 `GameState`、地图道路、军队坐标、动态战区或部署层。
- 不运行本机 Xcode / XCTest / 模拟器 / Probe / Smoke / Stage Regression / Dynamic Theater Regression / Full / 性能测试。

## 4. 架构依据

- Hex 仍是战术权威，官道状态来自 `HexTile.hasRoad`。
- 道路机动预判属于 `AppContainer` 给 `UnitInspectorView` 的只读派生数据，不应写回规则层。
- 本回合可达性以 `MovementRules.movementRange` 为准；敌控区口径以现有 `Faction.isHostile(to:)` 间接支持的 `MovementRules.isEnemyZoneOfControl` 为准。
- 真实移动仍必须通过 `Command.move -> RuleEngine -> CommandExecutor`。

## 5. 实现步骤

1. 在 `selectedUnitMobilityPreviewNotes` 内复用同一次 `MovementRules.movementRange` 结果，避免重复计算可达格数量。
2. 新增私有 `reachableRoadAccessNote`，筛选可达格里的官道 hex。
3. 若当前不在官道且本回合无法抵达官道，显示“本回合尚不能接入官道”。
4. 若可抵达官道，统计可达官道总格数、未受敌控区压迫的安全官道格数和受敌控区压迫的官道格数。
5. 保持既有基础/有效机动、武将官道加成、当前位置/郡县官道提示不变。
6. 更新核心文档、流程图、阶段索引和版本日志。

## 7. 文档更新要求

- `README.md`：同步 v2.4 武将道路/交战规则摘要。
- `md/flow/flow.md`：同步当前核心链路说明。

- `update_log.md`：追加 v2.4 版本记录，写明轻量检查和未跑重测试原因。

## 8. 验收标准

- 选中军队仍显示基础机动、有效机动、可达格数、武将官道加成和当前位置/郡县官道状态。
- 若本回合能进入官道，显示可达官道总数，并区分安全官道和敌控区压迫官道。
- 若当前不在官道且本回合不能进入官道，显示无法接入官道提示。
- 真实移动、路径、修路、敌控区、粮道和战略同步行为不变。
- 轻量检查通过；重测试交给云端 CI 或后续 Agent C artifact 复判。

## 9. 风险

- 本轮没有运行 UI 烟测，新增长文本在窄屏、长军队名和 Dynamic Type 下的实际换行仍需后续检查。
- 预判只统计本回合可达官道，不计算跨回合道路接入计划、完整补给线安全或修路收益。
- 敌控区压迫只沿用当前邻接敌军口径；完整外交借道、同盟通行和多势力通行制度仍待后续版本。
